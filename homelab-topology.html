04 — Troubleshooting Log

Field Diagnostic Entry

Documented diagnostic session: initial switch setup, management plane unreachable, ARP discovery failure, and resolution path. July 3, 2026.

TL-SG108PE · Initial Setup · Management UI Unreachable
July 3, 2026
Problem Statement

Newly acquired TP-Link TL-SG108PE (used, unknown prior configuration) was physically connected to the network — eero LAN port → switch Port 8, laptop → switch Port 6. The switch was successfully forwarding Layer 2 Ethernet frames and the host laptop had full internet connectivity routed through it. However, the switch management plane was completely unreachable. The default management IP (192.168.0.1) and the TP-Link fallback hostname (tplinkswitch.net) both failed. The eero app showed zero wired devices despite the physical connection being confirmed active.

Failure Points Identified
F1
Cable fault (suspected). First cable swapped — eero still not detecting the switch on Port 2. Ruled out cable as sole cause but confirmed cable integrity matters at this layer.
F2
Static IP conflict from prior config. The used switch had a previous owner's static IP configuration. Its management interface was sitting on an IP outside the current subnet, making it invisible to ARP and the eero DHCP table. The switch continued forwarding Layer 2 Ethernet frames independently of its management interface, demonstrating the separation between the switching plane and the management plane. A Layer 2 device operates on MAC address tables, not management IP reachability — which is why the laptop retained internet access while the switch remained undiscoverable.
F3
Factory reset incomplete on first attempt. Initial reset did not fully wipe prior configuration. A full power cycle (physical unplug, 30-second wait, cold boot) was required to force a clean state and trigger a fresh DHCP request.
F4
Laptop static IP mismatch. Early troubleshooting attempt set a manual static IP on the laptop to reach the switch's expected fallback address (192.168.0.1). This prevented the laptop from being on the eero's DHCP subnet and made ARP lookups unreliable. Resolved by returning ethernet adapter to DHCP before running ARP discovery.
Diagnostic Methodology

OSI Layer 1 through Layer 3 verification. Worked from the physical layer upward. Verified link lights on the switch before touching any software configuration. Swapped cables to eliminate physical layer variables. Confirmed eero was receiving traffic from the switch by checking the eero app's connected devices panel — "Multiple devices" appearing on Port 2 confirmed Layer 2 connectivity even when the management interface was invisible.

Used ipconfig to identify current subnet and gateway, then arp -a to enumerate devices on the network. Cross-referenced ARP physical addresses against the switch's MAC label to confirm identity.

Link light observation eero app · device panel ipconfig arp -a MAC label cross-reference Factory reset (pinhole) Power cycle
Resolution Path
R1
Replaced cable — eliminated physical layer variable.
R2
Factory reset via pinhole button — cleared prior owner's static IP configuration. Reset confirmed by simultaneous flash of all port LEDs.
R3
Full power cycle (physical unplug, cold boot) — forced switch to send a fresh DHCP discover to the eero. Switch was assigned 192.168.5.36 and appeared in eero device list as TL-SG108PE with correct MAC.
R4
Accessed management plane at http://192.168.5.36 — login successful with factory credentials. Changed admin password immediately. Set static IP in switch UI to lock management address permanently.
Outcome
  • Switch management accessible at static IP 192.168.5.36
  • Admin credentials changed from factory default
  • 802.1Q VLAN enabled · VLAN 10 configured for lab host segmentation · VLAN 20 reserved for future expansion with a VLAN-aware router/firewall
  • Port 6 PVID set to 10 — lab host traffic correctly assigned to LAB VLAN
  • Port 8 configured as router uplink · VLAN trunking reserved for future use with a VLAN-aware router/firewall
  • Port mirroring enabled — Port 7 as destination, Port 6 as source (ingress + egress)
  • Switch configuration saved to non-volatile memory
RESOLVED · Layer 2 config complete
